# PolicyMind 核心开发文档

本目录集中保存 PolicyMind 项目最初确定的三份核心文档。开发范围、代码接口和任务顺序发生冲突时，应按下面的职责划分判断。

## 文档导航

| 顺序 | 文档 | 主要内容 | 什么时候看 |
|------|------|----------|------------|
| 1 | [总体设计文档](./2026-07-01-policymind-design.md) | 产品定位、用户场景、总体架构、数据模型、Agent与RAG设计、验收指标 | 开始开发或判断功能是否偏离项目目标时 |
| 2 | [详细开发手册](./DEVELOPMENT_GUIDE.md) | 技术选型、模块边界、重要类型、核心函数、调用关系、安全与测试约束 | 编写具体模块和接口时 |
| 3 | [TASK 1-10实施计划](./2026-07-01-policymind-implementation.md) | TDD开发顺序、每个Task的文件、测试、命令与提交要求 | 实际执行开发任务和阶段验收时 |

## 三份文档的关系

```text
总体设计文档
    ↓ 决定“项目要解决什么问题”
详细开发手册
    ↓ 决定“各模块和重要函数怎么实现”
TASK 1-10实施计划
    ↓ 决定“按什么顺序开发和验收”
代码、测试与提交
```

## 冲突处理规则

1. 产品范围、业务目标和最终验收指标以总体设计文档为准。
2. 模块职责、类型定义、函数输入输出和安全规则以详细开发手册为准。
3. 开发步骤、红灯测试和提交顺序以TASK 1-10实施计划为准。
4. 审查发现的缺陷与后续修复，以 `docs/reviews/2026-07-02-task1-10/` 中的材料为准。
5. 修改任意核心约定时，应同步更新三份文档，避免代码和文档再次分叉。
